# Phala Protocol

**Status:** Draft v0.1.0
**Paper:** *Phala: Principal-Declared Welfare Feedback as a Protocol Extension for Autonomous Agent Networks*
**License:** Apache 2.0

Phala (Sanskrit for *fruit* or *the outcome of action*) is a protocol
extension for autonomous agent networks that closes two structural
gaps in existing agent protocols: the **outcome gap** (current
protocols have no vocabulary for whether a completed task served the
principal), and the **welfare inversion** (every existing feedback
mechanism defines a "good" outcome from the service provider's
perspective, not the principal's).

This repository contains the specification, a TLA+ safety-property
declaration (state machine in progress), and a Python reference
implementation of the five Phala primitives.

## The Five Primitives

| Primitive | What it records |
|---|---|
| `OutcomeEvent` | The objective facts of a task's resolution |
| `SatisfactionRecord` | The principal's quality signal for that outcome |
| `BeliefUpdate` | A scalar weight adjustment propagated back through the agent network |
| `PrincipalSatisfactionModel` | Per-context, principal-authored declaration of what *good* means |
| `WelfareTrace` | On-device longitudinal welfare signal that modulates network learning rate |

Together these primitives route outcome quality back to the agents that
produced it, governed by the principal's declared definition of a good
outcome, without exfiltrating the raw behavioral signals that produced
the update.

## Relationship to Other Protocols

Phala is an extension to A2A [1] and MCP [2]. It does not modify
either core protocol; all Phala primitives travel as extension fields
on existing messages or as new endpoints.

## Formal Properties

A correct Phala implementation MUST satisfy five safety properties:

- **OE-Uniqueness.** Every terminal task produces exactly one
  `OutcomeEvent`.
- **BU-Privacy.** No `BeliefUpdate` field is derived directly from
  `signal_components`. Only `weight_delta` is transmitted.
- **Chain-Monotonicity.** `SatisfactionRecord` entries are append-only;
  never modified or deleted.
- **Update-Boundedness.** All weights remain within declared
  [w_min, w_max] after any update.
- **PSM-Sovereignty.** When a valid PSM exists for a context, `valence`
  MUST be computed using the PSM's signal weights. No agent formula
  may override the principal's declaration.

The TLA+ specification of these properties lives in
[specification/Phala.tla](./specification/Phala.tla). The current
specification declares the invariants and state variables; a full state
machine suitable for TLC model-checking is planned for a subsequent
revision.

## Reproducible Simulations

The figures in the paper are generated by scripts in
[simulations/](./simulations/):

| Figure | Script | Claim verified |
|---|---|---|
| 1 | `valence_ref.py` | Sensitivity of the reference valence formula to each signal component |
| 2 | `participation_weight.py` | Harmonic decay gives d=7 agents 12.5% participation; exponential gives 0.8% |
| 3 | `welfare_adjustment.py` | Three 12-week welfare trajectories (improving, stable, degrading) |
| 4 | `welfare_adjustment.py` | Full parameter-space heatmap for `welfare_adjustment` |
| 5 | `travel_booking.py` | 20-episode routing convergence in a three-agent chain |

To reproduce:

```bash
python -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"
python simulations/participation_weight.py
python simulations/welfare_adjustment.py
python simulations/valence_ref.py
python simulations/travel_booking.py
```

## Reference Validators

The Python package under [src/phala/validators/](./src/phala/validators/)
implements the twelve invariants named in the paper (OE-1, OE-2, SR-1,
SR-2, BU-1 through BU-4, PSM-1, PSM-2, WT-1, WT-2). These are
deliberately small; enough to prove every invariant has working code
without committing to a full RPC implementation.

```bash
pip install -e ".[test]"
pytest
```
